## 〇、目标定位

**研发目标：任新田进厂简历 / 考研复试的项目展示。**

面试官看什么：
- 代码质量 & 架构设计能力
- 技术栈广度与深度
- 工程化与独立解决问题的能力
- 项目完整度（从代码到分发）

Star 数只是锦上添花，**技术深度才是主菜**。Origami 的 PlatformAdapter 插件架构 + Puppeteer 反爬 + sign-server 签名 + 完整工程化链路（CI/签名/安装器），足够撑起一场复试面试的项目展示。

## 一、合规红线（不可触碰）

### 绝对禁止

- ❌ **提供付费内容下载** — 绕过付费墙是违法的
- ❌ **做成 Web 服务 / API 对外开放** — 从「个人工具」变成「提供服务」，性质完全不同
- ❌ **内置第三方广告 / 付费功能** — 不靠这个赚钱
- ❌ **收集用户数据上传服务器** — Origami 是纯本地工具，永远不要加遥测/统计
- ❌ **在破解/灰产渠道推广** — 吾爱破解等平台 = 标签负面 + 法务风险（不是曝光渠道的问题，是社区调性跟你的目标背道而驰）

### 可以大胆做

- ✅ GitHub 正常发布 Release
- ✅ 技术博客（掘金、思否、CSDN） — 讲架构、讲技术方案
- ✅ 简历上写「多平台内容采集系统」
- ✅ 牛客/力扣/面试时主动展示项目
- ✅ 技术社群（V2EX、Reddit r/Python）分享设计思路

## 二、前车之鉴：erma0/douyin

### 发生了什么

- 项目：`erma0/douyin`（DouyinCrawler）
- ⭐ 1.6k Star，维护 7 年
- **2026年6月：因「合规审查」仓库被清空，所有代码移除**

### 踩了什么雷（推测）

1. **纯抖音工具** — 没有其他平台，抖音一掐直接死
2. **功能越界** — 10 种采集模式，覆盖了抖音几乎所有内容入口
3. **提供 Web 服务模式** — FastAPI + Web UI，可以部署成下载站
4. **知名度太高且集中在单一敏感领域** — 1.6k star，在抖音相关搜索中排名靠前
5. **没有合规缓冲** — 项目描述直白写「抖音爬虫」
6. **曝光渠道有问题** — 破解/灰色社区传播，吸引的非目标用户放大了风险

### 我们的教训

> erma0 的死因不是「被发现了」，而是「被发现时没有退路」。
> Origami 的 PlatformAdapter 就是这个退路。

### Origami 的优势

| 风险点 | erma0 | Origami |
|--------|-------|---------|
| 单一平台依赖 | ❌ 纯抖音 | ✅ PlatformAdapter 多平台 |
| Web 服务暴露 | ❌ FastAPI 可远程部署 | ✅ 纯本地 PyQt6 桌面端 |
| 功能越界 | ❌ 10 种模式全覆盖 | ✅ 下载为主，不过度 |
| 退路 | ❌ 无 | ✅ 关一个 adapter 即可 |
| 定位 | ❌ 抖音爬虫 | ✅ 多平台内容管理工具 |

## 三、对外展示策略

### 项目怎么说（简历 / 面试版）

> **Origami — 多平台内容采集与管理系统**
>
> 基于 Python + PyQt6 的 Windows 桌面应用，采用插件式平台适配器架构（PlatformAdapter），集成 Puppeteer 智能反爬引擎与独立 Node.js 签名服务。支持多平台内容获取与本地归档，具备托盘运行、单实例检测、代码签名、自动更新等完整工程化能力。
>
> 技术栈：Python 3.12 / PyQt6 / Node.js / Puppeteer / Inno Setup / SignPath

### README 措辞指南

**用什么词：**
- ✅ 「内容归档工具」
- ✅ 「学习实践」「个人研究」
- ✅ 「多平台内容管理」
- ✅ 「平台适配器架构」

**避免什么词：**
- ❌ 「爬虫」
- ❌ 「去水印」
- ❌ 「破解」「免费下载」
- ❌ 「抖音下载器」（定位太窄）

## 四、架构保护策略

### PlatformAdapter 是你的安全网

```
如果某一平台被要求下架：
  → 删除对应 adapter 文件
  → 项目继续以其他平台运行
  → 不会出现 erma0 式的「全盘清空」
```

### 建议

- 每个平台的 adapter 保持代码独立，不要交叉依赖
- 抖音 adapter 里不要写死其他平台的逻辑
- **尽快上线 B站 / 微博 adapter** — 从「抖音工具」变成「多平台工具」，合规风险大幅降低
- 如果未来某个平台被要求移除，**立即响应、干净移除**，不要拖

---

## 五、如果收到合规通知怎么办

1. **不要慌** — GitHub 会给时间响应
2. **先确认来源** — 是平台方（字节跳动）还是 GitHub 自动扫描
3. **立即移除对应 adapter** — 只删那个平台，项目主体不受影响
4. **回复说明已整改** — 如果确实违规，承认并修正是最好的策略
5. **更新 README 声明** — 加上合规声明
6. **不要对线** — 不要在 Issue/Social Media 上争吵，沉默处理
